Systems Development
Life Cycle

A four-phase documentation of the planning, analysis, design, and implementation of Wescue — a rescue-only dog adoption platform that matches adopters to shelter dogs using lifestyle data and algorithmic scoring.

Next.js 15 Supabase TypeScript Rescue-Only Platform July 2026
01

Phase 01 — Planning

Why should this system be built?

Define the business case, assess feasibility across four dimensions, and produce a project plan before any development begins.

Project Initiation

Approximately 6.3 million dogs enter U.S. shelters annually. Nearly 50% wait longer than optimal placement windows, and roughly 20% of adoptions are returned within the first year — the majority citing lifestyle incompatibility as the primary reason. Existing platforms like Petfinder aggregate listings but rely on adopters to self-screen, producing mismatches that overload shelter staff with repeat applicants and ultimately harm animal welfare outcomes.

Wescue addresses this by shifting the burden of compatibility assessment from adopter to platform. By collecting structured lifestyle data and running it through an algorithmic matching engine, the system surfaces the dogs most likely to succeed in a specific household — reducing both failed adoptions and the 45-minute per-application review burden on shelter coordinators.

The platform operates on a shelter-first, no-breeder mandate: every dog listed on Wescue must originate from a verified rescue organization.

Feasibility Analysis

Technical Feasibility Feasible

The development team has demonstrated working proficiency with the entire stack: Next.js 15 App Router, Supabase (PostgreSQL + Auth + Storage), TypeScript, and Tailwind CSS v4. The matching algorithm uses weighted scoring across eight compatibility dimensions — no novel AI research is required. All infrastructure components are proven, open-source or managed SaaS. A working prototype with 41 passing automated tests and 12 seeded dog listings confirms technical viability.

Economic Feasibility Feasible

CategoryItemEstimated Cost
DevelopmentSolo / small team (internal labor)$0
Hosting – LaunchVercel free tier + Supabase free tier$0/month
Hosting – GrowthVercel Pro + Supabase Pro (est. 500 shelters)~$75/month
Domainwescue.com annual registration~$15/year
Revenue PathShelter premium tier at $29/monthBreak-even at ~3 shelters

The platform reaches operational break-even with minimal paid shelter subscriptions. Grant funding from animal welfare foundations provides an additional non-commercial revenue path consistent with the nonprofit-aligned mission.

Organizational Feasibility Feasible

Wescue aligns directly with the core mandate of rescue organizations: getting the right dogs into the right homes quickly. Shelters are the primary data providers and the principal beneficiaries of better-matched applicants. Adopters gain a friction-reduced experience — the matching quiz requires zero typing and completes in under 60 seconds. No organizational change management is required beyond shelter onboarding. Potential resistance: shelters accustomed to their own intake forms may need a migration path, addressed in Phase 4.

Schedule Feasibility Achievable with Phasing

The MVP (authentication, matching engine, dog browse, apply flow) was delivered in approximately four weeks. Remaining features — shelter dashboard, email notifications, RescueGroups API integration — are scoped to subsequent two-week sprints. The platform provides value at each milestone and does not require full completion before launch.

Use Case Modeling

The system's primary actor is the Adopter — an authenticated or guest user seeking a rescue dog. Secondary actors include the Shelter Coordinator (future phase) and the Matching Engine (internal system actor).

UC-01
Find a Matching Dog
Actor: Adopter (guest or authenticated)
Goal: Receive a ranked list of compatible dogs
Flow: Open quiz → answer 8 lifestyle questions via buttons → optionally add notes → system scores all available dogs → top 5 results displayed with scores and reasons
Outcome: Ranked matches; adopter clicks a dog to view full profile
UC-02
Submit Adoption Application
Actor: Authenticated Adopter
Goal: Apply to adopt a specific available dog
Flow: View dog profile → click "Apply to Adopt" → modal opens → optionally add notes → submit → system validates availability + no duplicate → application stored
Outcome: Application record created (status: submitted); button changes to "Application Submitted ✓"
UC-03
Save Dog to Favorites
Actor: Authenticated Adopter
Goal: Save a dog for later consideration without applying
Flow: Click heart icon on any dog card or profile → system toggles favorite state → optimistic UI updates immediately → rollback on failure
Outcome: Dog appears in dashboard Saved Dogs page; removable at any time
UC-04
Complete Adopter Profile
Actor: Authenticated Adopter
Goal: Build a verified profile shelters review with applications
Flow: Dashboard → Your Profile → fill 5 sections (personal info, home details, home photos, ID upload, references) → save
Outcome: Profile upserted to database; photo ID and home photos stored in private Supabase Storage
UC-05
Track Application Status
Actor: Authenticated Adopter
Goal: Monitor where each application stands in the shelter's review process
Flow: Dashboard → Applications → list loads from API → each card shows dog, shelter, submission date, and current status badge
Outcome: Adopter sees real-time status; shelter notes are shown when provided

Phase 03 — Design

How will the system be built?

Specify the architecture, interface system, database schema, information architecture, and prototype fidelity levels before construction begins.

Design Strategy

OptionAssessmentDecision
Custom Development Full control over matching algorithm, adopter UX, and shelter data model. Core competitive advantage is the scoring engine — cannot be replicated by off-the-shelf software. Selected
Packaged Software No existing adoption platform offers lifestyle-based algorithmic matching. Available CMS or marketplace solutions would require extensive customization that eliminates cost savings. Rejected
Outsource Matching algorithm represents core IP. Outsourcing development risks loss of technical control and creates dependency for a system requiring continuous refinement. Rejected

Database Specifications

All tables reside in public schema on Supabase PostgreSQL. Row Level Security (RLS) is enabled on every table. Foreign keys cascade on delete to prevent orphaned records.

TablePrimary KeyKey ColumnsRLS Policy
sheltersuuidname, email, city, state, verifiedPublic read; admin write
dogsuuidshelter_id, name, breed, age, size, energy_level, good_with_*, status, photos[]Public read; shelter write (future)
applicationsuuiduser_id, dog_id, shelter_id, status, applicant_notes, shelter_notes · UNIQUE(user_id, dog_id)Owner read/insert/update
favoritesuuiduser_id, dog_id · UNIQUE(user_id, dog_id)Owner read/insert/delete
adopter_profilesuuiduser_id, personal info, home details, id_photo_url, home_photos (jsonb), personal_references (jsonb) · UNIQUE(user_id)Owner read/write
